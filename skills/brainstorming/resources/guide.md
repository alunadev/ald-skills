# Brainstorming Guide

## The Process

### 1. Understanding the Idea
*   **Check Context First:** Don't ask questions you can answer by reading the code.
*   **One Question Rule:** Only ask **one** question per message. If a topic needs deep exploration, break it down.
*   **Multiple Choice:** "Do you want A (faster) or B (more robust)?" is better than "How should we do this?"

### 2. Exploring Approaches
*   Always propose 2-3 approaches.
*   Discuss trade-offs (complexity vs. speed, scalability vs. YAGNI).
*   Lead with your recommendation but explain *why*.

### 3. Presenting the Design
*   **Chunking:** Never dump a massive design doc. Present 200-300 words at a time.
*   **Checkpoints:** Ask "Does this look right so far?" after each chunk.
*   **Coverage:** Ensure you cover Architecture, Components, Data Flow, Error Handling, and Testing.

## Key Principles
*   **YAGNI Ruthlessly:** Remove features that aren't strictly necessary for the MVP.
*   **Be Flexible:** If the user changes their mind, go back and revise.
*   **Incremental Validation:** Don't move to Planning until the Design is "locked."
